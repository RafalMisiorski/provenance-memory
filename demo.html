provenance-memory · ~180 lines, zero deps, MIT

Agent memory that updates a fact instead of hoarding every version of it.

When a fact changes, most agent memory keeps the old value in the index and hands it back later. This one keeps the current value, survives a restart, tells you what it replaced — and, unusually, has been broken on purpose by four independent adversaries until it stopped breaking.

The honest measurement

Sealed staleness test (deterministic scoring, no LLM judge), 63 changed facts across 4 seeds, local llama3.1 backend. Fail = a superseded value is returned when the current one is expected.

store & how it's usedstale-on-changed
mem0 — as-shipped (recall top-1) 77.8%
mem0 — used competently (you attach + sort by timestamp) 25.4%
provenance-memory (deterministic newest-wins) 0.0%
 The fair number is the middle row, not the top one: the 77.8% is partly a usage error (a naive top-1 recall). But even when you attach your own timestamps and sort by them, mem0 stays ~25% stale — because ~24% of the time it doesn't preserve the timestamp you gave it.
Caveats on the tin: local 8B backend, N=20×4, STALE mode only, wide CI [16–37%]. Directional and reproducible, not a three-decimal benchmark.

What it does — each line a passing test

  • ✓
    Change-detection, out-of-order safe. The value with the greatest logical timestamp is current. A late-arriving older write is recorded but never dethrones a newer one.
  • ✓
    Cross-session, across processes. Opt-in JSONL log with a stated contract: JSON-native values, corrupt-line tolerant, single-writer. Verified with two separate python invocations.
  • ✓
    Deterministic + auditable. recall() is an O(1) pure lookup. audit() returns value, when, session, source, revision, and what it replaced — correct even under out-of-order writes.
  • ✓
    Type-aware + provenance-keeping. 1, True, and 1.0 are different facts; a different source re-attesting a value is kept as corroboration, not dropped. No black box: no model, no API key, no vector DB, no deps.

How we know it's correct — the audit ledger

The first version passed its own 10 unit tests and was still wrong. So it was handed to independent adversaries — separate models, then a different vendor's model — whose only job was to break it. The cross-vendor round found three bugs four same-family rounds missed. Each defect — including ones the fixes themselves introduced — is now a named regression test.

audit 1 · model A 10 bugs in v1: shallow type-equality, arrival-order supersession, no persistence, corrupt-line crash, lying audit trail… 10 found
audit 2 · model B 7/10 fixed; found the fix reintroduced disk/memory divergence via an unvalidated field +1 HIGH
audit 3 · model B divergence fixed; found a depth-guard churning deep values + corroboration bloat +2
audit 4 · model B both fixed; found a dedup that dropped a newer write after an out-of-order one +1 HIGH
audit 5 · Codex (cross-vendor) a different vendor's model found 3 bugs 4 same-family rounds missed: UTF-8 bytes crashed load, a NaN line loaded instead of skipping, a deep value crashed on copy +3
audit 5 · re-verify fixes confirmed; caught one that was incomplete — corrupt bytes inside a quoted string still slipped through → strict per-line byte-decode +1
final · verify cross-vendor re-verify clean; 4,000 oracle-checked fuzz + 1,500 reload-determinism cases all pass 0 left

Where it is not magic (so you don't get sold something): on in-order, single-value-per-key facts this is equivalent to a ~15-line vector-store + newest-wins query. There is no capability moat. The pitch is the opposite of magic: your 15 lines would have had most of those 14 bugs too (out-of-order, mutation, persistence, corroboration). This is the version that already survived four adversaries, so you don't rediscover them in production. The deterministic-freshness idea is also known in the literature (arXiv 2606.01435).

Try it

# zero dependencies; runtime needs only the stdlib
python provenance_memory.py                 # live demo: recall / audit / out-of-order
python -m pytest test_provenance_memory.py  # 28 properties, ~0.3s

# in your agent:
from provenance_memory import ProvenanceStore
m = ProvenanceStore(path="mem.jsonl")       # omit path for pure in-memory
m.remember("refund_window_days", "30")
m.remember("refund_window_days", "7")       # supersedes
m.recall("refund_window_days")              # -> "7", every time, next process too
